# /dev-init Generator Skill Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Create two user-level skills — `/dev-init` (generates project-level dev workflow skills) and `/dev-init:improve` (audits and updates existing skills).

**Architecture:** Single monolithic SKILL.md per skill. The generator reads CLAUDE.md + filesystem, then writes three project-level skill files. The improver follows the claude-md-improver audit/report/propose/apply pattern.

**Tech Stack:** Claude Code skills (markdown), Bash for filesystem detection

**Spec:** `docs/superpowers/specs/2026-03-16-dev-init-generator-design.md`

**Reference templates:** The existing archetype project skills serve as the gold standard:
- `.claude/skills/dev/SKILL.md` — 250 lines, full feature workflow
- `.claude/skills/quick-fix/SKILL.md` — 127 lines, streamlined fixes
- `.claude/skills/cleanup/SKILL.md` — 148 lines, post-work checklist

## Chunk 1: /dev-init Generator Skill

### Task 1: Create the /dev-init skill directory

**Files:**
- Create: `~/.claude/skills/dev-init/SKILL.md`

- [ ] **Step 1: Create directory structure**

```bash
mkdir -p ~/.claude/skills/dev-init/improve
```

- [ ] **Step 2: Write the /dev-init SKILL.md**

Write `~/.claude/skills/dev-init/SKILL.md` with the following structure. This is the full skill content — it must contain all the logic since it's a monolithic skill.

**Frontmatter:**
```yaml
---
name: dev-init
description: Generate project-level development workflow skills (/dev, /quick-fix, /cleanup) tailored to this project's tech stack, commands, and conventions. Analyzes CLAUDE.md and filesystem to produce skills with exact commands baked in.
---
```

**Skill body must include these sections in order:**

1. **Overview** — One paragraph explaining what the skill does: analyzes project, generates three skills, supports interactive/auto modes.

2. **Phase 1: Pre-flight** — Instructions to check for existing `.claude/skills/dev/SKILL.md`, `.claude/skills/quick-fix/SKILL.md`, `.claude/skills/cleanup/SKILL.md`. If any exist, list them and ask user: overwrite all, selectively, or abort. Do not silently overwrite.

3. **Phase 2: CLAUDE.md gate** — Check if `CLAUDE.md` exists in project root. If not, run the `/init` command (Claude Code built-in) to generate one. After `/init` completes or if CLAUDE.md already exists, proceed.

4. **Phase 3: Project analysis** — Detailed instructions to extract project details from two sources:

   **From CLAUDE.md** (read the file):
   - Project name and description
   - Build/test/dev/lint commands
   - Architecture overview
   - Key conventions and gotchas

   **From filesystem** (run detection commands):
   ```bash
   # Language/runtime detection
   ls pyproject.toml package.json Cargo.toml go.mod Makefile 2>/dev/null

   # Framework detection (read manifest dependencies)
   # For Python: grep for fastapi, django, flask in pyproject.toml
   # For Node: grep for next, express, react in package.json
   # For Rust: grep for actix, axum in Cargo.toml

   # Structure detection
   ls -d prds/ docs/ specs/ scratchpad/ supabase/ tests/ test/ __tests__/ 2>/dev/null
   ls Dockerfile docker-compose.yml .github/workflows/ .circleci/ 2>/dev/null
   ls .eslintrc* ruff.toml .rubocop.yml biome.json 2>/dev/null

   # Derive project name from directory
   basename $PWD
   ```

   **Derived values table** — include the full table from the spec:
   - Worktree prefix: `../{project-name}-feature/` and `../{project-name}-fix/`
   - Branch convention: `feature/{description}` and `fix/{description}`
   - Health check: framework-dependent (FastAPI → `/health`, Express → `/health`, Rails → `/up`)
   - Port: from CLAUDE.md or framework default
   - CLAUDE.md wins when filesystem detection disagrees

5. **Phase 4: Generate skills** — Instructions to write three files. This section must contain the **complete skill templates** with clear markers for where project-specific values go. Structure it as three subsections:

   **4a. Generate /dev skill** — Write `.claude/skills/dev/SKILL.md` following the structure of the reference template (archetype's existing dev skill). Include:
   - Provenance header: `<!-- Generated by /dev-init on {date}. Safe to customize — /dev-init:improve will propose updates, not overwrite. -->`
   - Frontmatter with name and description
   - Mode Selection section (universal — copy exactly from reference)
   - Roles section (universal)
   - 18 workflow steps where:
     - Steps with commands use the detected project-specific values (test command, dev server, health check, worktree paths)
     - Conditional steps are only included if the relevant infrastructure was detected in Phase 3
     - Project-specific notes from CLAUDE.md conventions/gotchas are included in the Implement step
   - Key Rules section (universal — copy exactly from reference)

   **4b. Generate /quick-fix skill** — Write `.claude/skills/quick-fix/SKILL.md` following the reference template. Include:
   - Provenance header
   - Frontmatter
   - Mode Selection (universal)
   - When to Use / When NOT to Use (universal, but add project-specific "when not to use" items from CLAUDE.md if applicable)
   - 7 workflow steps with project-specific commands
   - Key Rules (universal)

   **4c. Generate /cleanup skill** — Write `.claude/skills/cleanup/SKILL.md` following the reference template. Include:
   - Provenance header
   - Frontmatter
   - Mode Selection (universal)
   - When to Use (universal)
   - 10 checklist items with project-specific commands, process names, ports, and database tooling
   - Post-Cleanup State checklist (universal)

   **Critical instruction for the generator:** The skill must tell Claude to write the full content of each generated skill — not placeholders or abbreviated versions. Each generated skill should be 100-250 lines, fully self-contained, with every workflow step written out. Use the archetype reference skills as the model for length, detail level, and tone.

   **Conditional step logic:** Include a decision table mapping detected infrastructure to which steps to include/exclude:

   | Detection | Include in /dev | Include in /quick-fix | Include in /cleanup |
   |-----------|----------------|----------------------|---------------------|
   | PRD directory found | Steps 2-3, 13, 15 | — | — |
   | `scratchpad/` exists | Steps 16-17 | Step 7 | Steps 8-9 |
   | Dockerfile found | Docker build note in step 6 | — | Docker in step 5 |
   | Database tooling found | DB note in step 6 | — | DB state check (step 6) |
   | Linter config found | Lint in step 8 | Lint in step 4 | — |
   | No tests detected | Step 8-9 with "skip until tests added" note | Step 4 with note | Step 10 with note |
   | No git remote | Skip PR steps (7, 11) | Skip PR steps (5-6) | Skip step 3 |

6. **Phase 5: Verification** — Output a summary table showing what was generated:
   ```
   | Skill | Path | Key Commands |
   |-------|------|-------------|
   | /dev | .claude/skills/dev/SKILL.md | test: `uv run pytest`, dev: `uv run uvicorn ...` |
   | /quick-fix | .claude/skills/quick-fix/SKILL.md | test: `uv run pytest` |
   | /cleanup | .claude/skills/cleanup/SKILL.md | processes: uvicorn, ports: 8000 |
   ```
   Remind user they can customize the generated files and run `/dev-init:improve` later.

7. **Edge cases** — Include the full edge case table from the spec as instructions for the generator to follow.

- [ ] **Step 3: Verify the skill file is well-formed**

Read back `~/.claude/skills/dev-init/SKILL.md` and verify:
- Frontmatter is valid YAML between `---` markers
- All 5 phases are present and in order
- Phase 4 contains complete template guidance for all three skills
- The conditional step table is present
- Edge cases are documented
- The file is self-contained (no external file references needed)

- [ ] **Step 4: Commit**

```bash

```

## Chunk 2: /dev-init:improve Auditor Skill

### Task 2: Create the /dev-init:improve skill

**Files:**
- Create: `~/.claude/skills/dev-init/improve/SKILL.md`

- [ ] **Step 1: Write the /dev-init:improve SKILL.md**

Write `~/.claude/skills/dev-init/improve/SKILL.md` following the claude-md-improver pattern. Structure:

**Frontmatter:**
```yaml
---
name: improve
description: Audit existing /dev, /quick-fix, and /cleanup skills against current CLAUDE.md and filesystem. Reports stale commands, missing capabilities, and invalid paths. Proposes targeted updates with diffs.
---
```

**Skill body must include these sections:**

1. **Overview** — Audits and updates existing project-level dev skills. Follows the claude-md-improver pattern: discover → assess → report → propose → apply.

2. **Phase 1: Discovery** — Find existing skills:
   ```bash
   ls .claude/skills/dev/SKILL.md .claude/skills/quick-fix/SKILL.md .claude/skills/cleanup/SKILL.md 2>/dev/null
   ```
   Check for provenance header (`<!-- Generated by /dev-init`). If no provenance header found, warn that these may be hand-written skills and ask user whether to proceed.

   If no skills found, tell user to run `/dev-init` first.

3. **Phase 2: Assessment** — For each skill found, check:

   | Check | Method |
   |-------|--------|
   | Command accuracy | Extract commands from skill, compare against CLAUDE.md commands section |
   | Path validity | Extract referenced paths (PRD dirs, scratchpad, debt), verify they exist on filesystem |
   | New capabilities | Read CLAUDE.md for commands/conventions not present in skills |
   | Removed capabilities | Find commands in skills that are no longer in CLAUDE.md or filesystem |
   | Step relevance | Check conditional step triggers (Docker, Supabase, linter, etc.) against current filesystem |
   | Port/URL accuracy | Compare hardcoded ports and health check URLs against CLAUDE.md |
   | Worktree paths | Verify project name in worktree paths matches `basename $PWD` |

4. **Phase 3: Report** — Output the audit report. Format:
   ```
   ## Dev Skills Audit Report

   ### /dev skill
   - [OK] Test command: `uv run pytest` matches CLAUDE.md
   - [STALE] Dev server port: skill says 8000, CLAUDE.md says 3000
   - [MISSING] Lint command `uv run ruff check` in CLAUDE.md not in skill
   - [INVALID] References `scratchpad/debt/` but directory doesn't exist

   ### /quick-fix skill
   ...

   ### /cleanup skill
   ...

   ### Summary
   - X checks passed
   - Y issues found (Z stale, W missing, V invalid)
   ```

   **Always output the report BEFORE proposing changes.** Let user see the full picture first.

5. **Phase 4: Propose updates** — For each issue found, show:
   - Which skill file to update
   - The specific change as a diff block
   - Why this change is needed

   Format:
   ```markdown
   ### Change 1: Update dev server port in /dev skill

   **Why:** CLAUDE.md now specifies port 3000, but skill still says 8000.

   **File:** `.claude/skills/dev/SKILL.md`
   ```diff
   - uv run uvicorn input_engine.main:app --port 8000 --reload
   + uv run uvicorn input_engine.main:app --port 3000 --reload
   ```
   ```

   Ask user: "Apply all changes, apply selectively, or skip?"

6. **Phase 5: Apply** — After user approval, use the Edit tool to make the changes. Do not rewrite entire files — make targeted edits only.

- [ ] **Step 2: Verify the skill file is well-formed**

Read back `~/.claude/skills/dev-init/improve/SKILL.md` and verify:
- Frontmatter is valid
- All 5 phases present
- Assessment criteria table is complete
- Report format is concrete
- Diff format for proposals is shown
- Instructions are clear enough to follow without external context
